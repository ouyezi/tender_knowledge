## User Scenarios & Testing *(mandatory)*

### User Story 1 - 产品分类树管理 (Priority: P1)

知识库管理员需要在「产品分类中心」维护产品线/业务线分类树（如餐补、体检、电影卡），
以便后续导入、检索和推荐都能按统一产品口径过滤。

**Why this priority**: Product Category 是 V3.0 一等分类对象，Epic 1 文件用途确认、
Epic 2/3 模板与标书标记、Epic 5 检索过滤均依赖此底座；无产品分类则后续 Epic 无法启动。

**Independent Test**: 管理员可在产品分类中心独立完成「创建三级分类 → 维护别名 →
查询分类树与详情」全流程，无需章节分类或其他 Epic 能力即可交付可用价值。

**Acceptance Scenarios**:

1. **Given** 空的知识库产品分类中心，**When** 管理员创建根分类「福利产品」及子分类
   「餐补」「体检」，**Then** 分类树按层级展示，且每个节点含名称、编码、描述与启用状态。
2. **Given** 已存在分类「餐补」，**When** 管理员为其添加别名「员工餐补」「餐饮福利」，
   **Then** 按标准名或任一别名搜索均可定位该分类。
3. **Given** 已有多级分类树，**When** 管理员查询某分类详情，
   **Then** 返回父级路径、子节点列表、别名列表、状态及创建/更新时间。
4. **Given** 管理员编辑分类名称或描述，**When** 保存成功，
   **Then** 变更即时生效且可被后续读取接口获取（供 Epic 1 导入确认页选用）。

---

### User Story 2 - 章节分类体系管理 (Priority: P2)

知识库管理员需要在「章节目录中心」维护 Chapter Taxonomy（章节类型标准名与同义章节名），
并将章节分类绑定到一个或多个产品分类，以归一化不同标书中的章节命名差异。

**Why this priority**: 章节语义归一是检索、模块组织建议与生成辅助的核心过滤维度；
与 P1 并列构成 Epic 0 双底座，但可独立于分类生命周期（P3）先交付基础维护能力。

**Independent Test**: 管理员可独立完成「创建章节类型 → 维护同义名 → 绑定产品分类 →
浏览层级关系」全流程，不依赖文件导入或 Bid Outline 数据。

**Acceptance Scenarios**:

1. **Given** 空的知识库章节分类，**When** 管理员创建标准章节类型「售后服务方案」
   及同义名「驻场服务方案」「服务保障方案」，**Then** 系统将其视为同一章节语义，
   按任一同义名均可检索到该类型。
2. **Given** 已存在章节类型「供应链介绍」，**When** 管理员将其绑定到产品分类
   「餐补」和「物资采购」，**Then** 绑定关系可被查询，且支持按产品分类筛选章节类型列表。
3. **Given** 管理员设置章节类型的父子层级（如「方案类」→「售后服务方案」），
   **When** 保存后查看分类树，**Then** 层级关系正确展示且无循环引用。
4. **Given** Epic 2/3 尚未上线，**When** 管理员手动维护章节类型，
   **Then** 系统不要求 Bid Outline 或 Template Chapter 数据即可完整使用维护功能。

### User Story 3 - 分类生命周期与影响分析 (Priority: P3)

知识库管理员在停用、合并或归档分类前，需要查看该分类关联的知识对象数量，
并安全执行状态变更，避免破坏已有资产引用或造成数据孤岛。

**Why this priority**: 分类治理的「可安全变更」是长期运营必需能力；依赖 P1/P2 已有
分类数据，但对 Epic 1 最小启动非阻塞（Epic 1 仅需读取启用中的分类）。

**Independent Test**: 管理员选择任一已使用的分类，查看影响分析报告，并完成停用或
合并操作，全程可验证关联计数与目标状态正确。

**Acceptance Scenarios**:

1. **Given** 分类「餐补」已被若干知识对象引用（含 KU、模板、目录、候选知识等类型），
   **When** 管理员发起停用或合并前查看影响分析，**Then** 系统按对象类型展示关联数量，
   且总数与明细一致。
2. **Given** 管理员将分类 A 合并入分类 B，**When** 合并确认完成，
   **Then** 原引用 A 的对象改为引用 B，A 进入不可再被选用的终态，且操作可追溯。
3. **Given** 管理员将分类设为「归档」，**When** 归档完成，
   **Then** 该分类不再出现在默认可选列表中，但历史引用与审计记录仍可查询。
4. **Given** 父分类下有启用中的子分类，**When** 管理员尝试停用父分类，
   **Then** 系统阻止操作或要求一并处理子分类，并给出明确提示。

---

### Edge Cases

- 创建分类时使用与现有节点重复的标准名或编码：系统拒绝并提示冲突项。
- 为同一分类添加已属于其他分类的别名：系统拒绝并说明占用情况。
- 合并源分类与目标分类存在父子关系：系统阻止并提示需调整层级后再合并。
- 停用/归档仍被大量对象引用的分类：允许操作但影响分析 MUST 展示完整计数；
  合并后引用迁移 MUST 保持可追溯。
- 删除请求：已发布或曾被引用的分类 MUST NOT 物理删除，仅支持停用/归档/合并。
- 空知识库首次使用：产品分类中心与章节目录中心均可从空状态开始维护，无需预置数据。
- 跨知识库隔离：一知识库内的分类 MUST NOT 被另一知识库的管理操作修改或可见。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持在知识库内创建、编辑、查询多级 Product Category 树。
- **FR-002**: 每个 Product Category MUST 包含名称、编码、别名列表、描述、状态
  （启用/停用/归档）、父级引用及时间戳。
- **FR-003**: 系统 MUST 支持 Product Category 别名与同义词的增删改查，且别名在
  知识库内唯一。
- **FR-004**: 系统 MUST 支持 Chapter Taxonomy 标准名、同义章节名及层级关系的维护。
- **FR-005**: 系统 MUST 支持将 Chapter Taxonomy 绑定到一个或多个 Product Category，
  并支持按产品分类筛选章节类型。
- **FR-006**: 系统 MUST 在停用、合并或归档分类前提供影响分析，按对象类型展示关联
  KU、Wiki、Template、Template Chapter、Bid Outline、Manual Asset、候选知识等数量。
- **FR-007**: 系统 MUST 支持 Product Category 与 Chapter Taxonomy 的合并操作，
  合并后引用迁移至目标分类且源分类不可再被选用。
- **FR-008**: 系统 MUST 支持分类状态管理：启用、停用、归档；已引用分类 MUST NOT
  物理删除。
- **FR-009**: 系统 MUST 提供分类树、分类详情、创建、更新、停用/归档、别名维护、
  影响范围查询等管理能力，供管理后台与后续 Epic 消费。
- **FR-010**: 系统 MUST 记录分类创建、编辑、合并、停用、归档等操作的审计信息
  （操作者、时间、变更摘要），满足全链路可追溯原则。
- **FR-011**: 当存在自动分类建议时（Epic 1 及以后），用户人工选择的分类结果 MUST
  优先于系统建议并持久化；Epic 0 MUST 在数据模型与业务规则层面预留该优先级。
- **FR-012**: 系统 MUST NOT 在本 Epic 内实现：文件内容自动训练分类模型、目录级检索
  推荐、候选知识确认工作台、角色权限体系。
- **FR-013**: 从 Bid Outline / Template Chapter 自动发现候选章节类型的完整流程 MUST
  留待 Epic 2/3；Epic 0 仅交付章节类型手工维护及与产品分类的绑定能力。

### Key Entities

- **Product Category**: 产品线/业务线/标书适用产品分类；多级树结构；可被 KU、Wiki、
  Template、Template Chapter、Bid Outline、Manual Asset 关联；含别名与生命周期状态。
- **Chapter Taxonomy**: 章节语义标准类型；含标准名、同义章节名、层级；可绑定多个
  Product Category；用于归一化不同标书章节命名。
- **Knowledge Base**: 分类的最高隔离单位；所有分类操作在单一知识库范围内生效。
- **Classification Impact Report**: 停用/合并/归档前的关联统计视图；按对象类型
  汇总引用数量，供管理员决策。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员可在 5 分钟内完成「创建三级产品分类并添加至少 2 个别名」，
  且后续查询可按别名定位该分类。
- **SC-002**: 管理员可在 5 分钟内完成「创建章节类型、添加 2 个同义名并绑定 2 个
  产品分类」的完整维护流程。
- **SC-003**: 对任一已被引用的分类，影响分析在 3 秒内展示按对象类型分组的关联数量
  （在常规数据量下，单分类关联对象 ≤ 10,000 条）。
- **SC-004**: Epic 1 导入确认页所需的产品分类与章节类型列表，100% 可从本 Epic
  交付的查询能力获取，无需手工维护平行字典。
- **SC-005**: 分类合并后，原引用对象的分类归属迁移成功率 100%，且管理员可在审计记录
  中追溯合并操作。
- **SC-006**: 95% 的分类维护操作（创建、编辑、别名、绑定）首次尝试成功，无需因
  冲突或层级错误重复提交超过 1 次。

## Assumptions

- 目标用户为知识库管理员（运营/治理人员），非投标业务人员；V3.0 暂不区分角色权限，
  所有登录用户均可执行分类管理（与总需求 §3 权限暂缓一致）。
- 本 Epic 以标书知识库为首个落地场景；Product Category 预置示例（餐补、体检等）
  可由管理员手工创建，不要求系统内置行业模板包。
- industry、project_type、customer_type 作为二级过滤维度，不在 Epic 0 实现，
  仅 Product Category 与 Chapter Taxonomy 纳入底座范围。
- 自动发现候选章节类型（来自 Bid Outline / Template Chapter）的流水线在 Epic 2/3
  接入；Epic 0 保证 taxonomy 数据结构与绑定关系可被后续流程写入。
- 影响分析统计的对象类型与 Epic 1–3 将产生的关联一致；Epic 0 上线初期计数可能为
  零，但报告结构与接口 MUST 就绪。
- 单知识库内的分类编码在同级 siblings 下唯一；跨层级允许不同分支使用相同编码
  （若业务后续要求全局唯一，在 plan 阶段确认）。
