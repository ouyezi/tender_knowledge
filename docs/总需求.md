## 1. 产品概述

### 1.1 产品定位

企业知识库平台是一个企业级知识资产生产、治理与服务平台，面向标书、产品、资质、案例、制度等多业务场景，将企业历史文档、标书模板库、实际标书、人工资产、标准内容和产品分类统一治理，形成可检索、可复用、可追溯、可用于 AI 生成的知识资产底座。

V3.0 在 V2.0 的基础上，进一步面向真实标书素材体系升级。当前版本不做目录或文件夹批量导入，所有来源均按“一个文件一次导入”的方式进入系统。平台需要理解和治理以下输入：

- 标书模板文件：从历史模板库中逐个导入的模板文件，主要作为知识来源和章节小模块组织建议来源。
- 产品分类：餐补、礼包卡、电影卡、保险、体检、物资采购、在线采购平台等业务线或产品线。
- 实际标书目录：历史标书或目标标书的多级目录结构。
- 标书模板：可拆解为章节模块、固定段落、变量、条件章节和引用规则，用于提供组织建议，而不是决定最终标书结构。
- 实际标书正文：作为知识来源导入，经解析、提取、人工确认后进入正式知识库。
- 外部招标约束：招标文件中的标书结构要求、评分点、废标项、格式要求、响应要求等不在知识库内治理，但会作为检索、推荐和生成辅助的外部主输入。

### 1.2 V3.0 核心升级目标

V3.0 的目标是将平台从“通用知识生产平台”升级为“真实标书素材资产平台”。

新增目标：

1. 支持单文件导入，文件可被标记为实际标书、标书模板、资质材料、PPT 素材、封面攻略、写作攻略等用途。
2. 支持标书模板文件解析，将模板中的章节、段落和素材片段转化为可治理的 Template、Template Chapter 和 Candidate Knowledge。
3. 支持产品分类体系，将产品分类贯穿知识、模板、目录、素材推荐和生成辅助。
4. 支持实际标书目录抽取，将历史标书目录沉淀为 Bid Outline 和 Chapter Pattern。
5. 支持候选知识确认工作台，实际标书和模板导入后先进入候选区，人工确认后再入库。
6. 支持基于外部招标约束进行知识推荐，模板库只提供小模块组织建议和历史表达参考。
7. 支持目录级检索和模块组织建议，基于外部目标目录匹配历史目录、章节模式和模板库片段。

### 1.3 继承 V2.0 的核心原则

V3.0 继续遵守 V2.0 的核心原则：

- 平台优先：标书知识库是首个场景，底层模型支持多业务扩展。
- 知识资产优先：核心消费对象是 KU、Wiki、Template，不是原始文件。
- 章节优先：以章节结构作为知识组织和生成边界。
- 高度自动化：LLM 辅助解析、分类、提取、推荐，人工负责确认与治理。
- 全链路可追溯：任何知识和生成内容均可追溯到来源文件、目录节点、版本和导入记录。
- 检索优先于生成：先保证找得准、找得全、可追溯，再生成。
- 人工确认优先：实际标书和模板导入后的候选结果必须由人工确认后才进入正式库。
- 人工编辑优先：Document Tree、Bid Outline、Template Chapter 经人工修正后，后续自动重解析不得直接覆盖，只能生成待确认差异。
- 招标约束优先：实际标书应用生成时，以招标文件提供的结构要求、评分点、废标项等外部约束为最高优先级；模板库不得覆盖或替代招标约束。

### 1.4 V2.0 继承范围

V3.0 是对 V2.0 的增量升级，不废弃 V2.0 的基础能力。以下 V2.0 章节在 V3.0 中继续有效：

- Knowledge Base 与知识库策略。
- Document、Document Tree、Knowledge Unit、Manual Asset、Template、Wiki 的基础定义。
- 文档异步解析流水线。
- KU 提取、自动分类、审核发布、版本管理、检索控制。
- 混合检索、Knowledge Pack、素材推荐。
- AI 生成辅助、冲突检测、生成引用绑定。
- 任务中心、评测看板、开放 API、安全、可观测性和容错要求。

若 V3.0 对同一对象或流程给出新增字段或新规则，以 V3.0 为准；未覆盖部分沿用 V2.0。

### 1.5 产品边界

平台负责：

- 单文件导入与文件用途识别。
- 标书模板文件解析与治理。
- 实际标书文档解析、目录抽取、知识提取。
- 产品分类、章节分类、知识分类管理。
- 候选知识确认、审核、发布。
- 模板库模块化治理、模块组织建议和章节结构参考。
- KU/Wiki/Template/Manual Asset 检索与素材推荐。
- 基于外部招标约束、知识包和模板库参考模块的章节草稿辅助。
- 来源追溯、版本、审计、任务、评测。

平台不负责：

- 完整投标项目管理。
- 标书最终排版和 PDF 交付。
- Word 类多人实时协同编辑。
- 自动替用户完成所有章节取舍。
- 自动解决业务冲突或合规冲突。
- 基于中标结果的自动策略决策，V3.0 仅预留效果评估数据结构。
- 招标文件完整解析系统、评分规则管理系统、废标项判定系统；知识库只消费这些系统或人工输入提供的结构化结果。

## 2. 总体架构

### 2.1 四层资产体系

V3.0 将 V2.0 的三层体系扩展为四层：

```text
来源层：Imported File / Document / Manual Asset
  ↓
结构层：Document Tree / Bid Outline / Template Library / Template Chapter
  ↓
知识层：Knowledge Unit / Wiki / Chapter Pattern / Product Category
  ↓
服务层：Retrieval / Recommend / Module Suggestion / Generation Assist / Open API
```

外部约束层不属于知识库内部资产模型，但在服务层调用时作为最高优先级输入：

```text
Tender Requirement Context（外部输入）
  = 标书结构要求 + 评分点 + 废标项 + 格式要求 + 响应要求
```

### 2.2 知识生产路径

V3.0 支持六条知识生产路径：

| 路径 | 来源 | 中间对象 | 产出 |
|------|------|----------|------|
| 文档知识生产 | 实际标书、历史方案、项目文档 | Document Tree | KU / Wiki |
| 模板知识生产 | 标书模板文件 | Template / Template Chapter / Template Material | Candidate Knowledge / KU / Wiki |
| 目录沉淀 | 历史实际标书目录、模板目录 | Bid Outline | Chapter Pattern / Template Draft |
| 人工资产生产 | 资质、证照、授权书 | Manual Asset | Manual Asset / KU |
| 标准内容生产 | 人工编写、LLM 辅助 | Wiki | Wiki / KU |
| 生成回写 | 生成草稿 | Generated Draft | Candidate Knowledge / KU |

模板文件映射规则：

```text
标书模板文件 → Imported File
  → Document / Template
  → Template Chapter / Template Material
  → Candidate Knowledge
```

V3.0 不解析文件夹层级，不通过文件夹自动生成 Template Library。若需要保留“模板库”概念，由用户在导入单个模板文件时手工选择或创建 Template Library 作为归类标签。

### 2.3 V3.0 主链路

```text
单文件导入
  → 文件用途识别与人工确认
  → 模板库解析 / 文档解析 / 资质识别
  → 目录抽取 / 文档树生成
  → 候选知识生成
  → 人工确认
  → 发布为 KU / Wiki / Template / Manual Asset / Chapter Pattern
  → 面向外部招标约束的检索推荐 / 模块组织建议 / 章节草稿辅助
```

### 2.4 最小闭环

V3.0-MVP 对应 Phase 1 至 Phase 3，最小闭环：

```text
导入一个实际标书或模板文件
  → 系统识别文件类型和用途
  → 解析文档内容
  → 生成 Bid Outline 与 Candidate Knowledge
  → 人工确认产品分类、章节类型、知识类型
  → 发布 KU / Template Chapter
  → 按产品分类和目标章节检索推荐
```

目录级模块组织建议、Template Instance 高级预留、章节草稿生成升级属于 V3.0-增强或 V3.0-高级，不纳入 MVP 必达范围。

## 3. 权限设计暂缓说明

V3.0 暂不设计用户角色和权限体系。当前版本默认在单一受信任使用环境中运行，只关注知识生产、知识治理、检索优化和生成辅助能力。

当前版本仅保留以下最小要求：

- 所有导入、确认、发布、废弃、生成和检索反馈操作需要记录操作日志。
- 数据对象保留 `created_by`、`updated_by`、`reviewed_by` 等字段作为审计预留，但不做权限判断。
- API 和页面不按角色区分能力边界。
- 角色体系、权限矩阵、字段级权限、外部审计角色等均作为后续版本能力预留。

## 4. 多知识库与分类体系

### 4.1 Knowledge Base

知识库仍是最高隔离单位。V3.0 标书知识库需要预置以下中心：

- 标书模板库中心。
- 实际标书中心。
- 产品分类中心。
- 章节目录中心。
- 资质材料中心。
- 标准 Wiki 中心。
- 候选知识中心。
- 检索与推荐中心。

### 4.2 Product Category

Product Category 是 V3.0 新增的一等分类对象，用于表达产品线、业务线或标书适用产品。

示例：

- 餐补。
- 礼包卡。
- 乐福卡。
- 电影卡。
- 保险。
- 体检。
- 物资采购。
- 在线采购平台。

字段模型：

```json
{
  "category_id": "",
  "kb_id": "",
  "parent_id": "",
  "category_name": "",
  "category_code": "",
  "aliases": [],
  "description": "",
  "status": "active | inactive | archived",
  "created_time": "",
  "updated_time": ""
}
```

要求：

- 支持多级分类。
- 支持同义词和别名。
- 支持分类合并、停用、归档。
- KU、Wiki、Template、Template Chapter、Bid Outline、Manual Asset 均可关联产品分类。
- 检索、推荐、模块组织建议、生成辅助均支持按产品分类过滤。

### 4.3 Chapter Taxonomy

Chapter Taxonomy 是章节分类体系，用于归一化不同标书中的章节命名。

示例：

- 企业综合实力。
- 系统对接方案。
- 售后服务方案。
- 增值服务方案。
- 资金风险管控。
- 商品管理。
- 供应链介绍。
- 产品功能介绍。
- 管理后台功能。

要求：

- 支持章节类型标准名。
- 支持同义章节名，例如“售后服务方案”“驻场服务方案”“服务保障方案”。
- 支持章节层级关系。
- 支持与 Product Category 绑定。
- 支持从 Bid Outline 和 Template Chapter 中自动发现候选章节类型。

### 4.4 分类体系关系

V3.0 中存在多类分类维度，各自职责如下：

| 分类维度 | 作用 | 示例 |
|----------|------|------|
| Product Category | 产品线/业务线过滤与模块建议 | 餐补、体检、电影卡 |
| Chapter Taxonomy | 章节语义归一化 | 供应链介绍、售后服务方案 |
| knowledge_type | KU 内容类型 | 方案、案例、资质、产品 |
| industry | 行业维度 | 政企、金融、教育 |
| project_type | 项目类型 | 采购、服务、平台建设 |
| customer_type | 客户类型 | 政府、企业、集团客户 |

优先级规则：

- 模块组织建议优先使用 Product Category 和 Chapter Taxonomy。
- 内容检索优先使用 knowledge_type、Product Category 和 Chapter Taxonomy。
- 合规素材推荐优先使用 Manual Asset 类型、Product Category 和有效期。
- industry、project_type、customer_type 作为二级过滤条件。
- 所有分类建议均可由人工覆盖，人工结果优先于自动识别结果。

## 5. 单文件导入模型

### 5.1 File Import

File Import 表示一次单文件导入操作。V3.0 不支持目录导入、文件夹扫描、批量导入和目录结构解析。

字段模型：

```json
{
  "import_id": "",
  "kb_id": "",
  "file_name": "",
  "file_type": "docx | pdf | ppt | xlsx | image | other",
  "file_size": 0,
  "file_hash": "",
  "storage_path": "",
  "file_purpose": "actual_bid | template_file | qualification | ppt_material | cover_guide | writing_guide | wiki_source | other",
  "product_category_ids": [],
  "chapter_taxonomy_id": "",
  "status": "uploaded | need_confirm | processing | completed | failed | ignored",
  "created_by": "",
  "created_time": "",
  "updated_time": ""
}
```

要求：

- 每次只导入一个文件。
- 文件导入后必须进入用途确认。
- 文件用途确认后才进入后续解析或候选知识生成。
- 不保留文件夹路径，不从文件夹名推断分类。
- 文件名可用于辅助推断产品分类、章节类型和素材用途，但必须允许人工覆盖。

### 5.2 文件用途确认

导入后系统展示文件用途确认界面。

用户可确认：

- 文件用途：实际标书、标书模板、资质材料、PPT 素材、封面攻略、写作攻略、Wiki 来源、其他。
- 产品分类。
- 章节类型。
- 是否进入解析。
- 目标对象类型：Document、Template Material、Manual Asset、Wiki、忽略。

验收标准：

- 用户上传一个文件后，系统生成 File Import 记录。
- 系统根据文件名和内容摘要给出用途建议。
- 用户确认用途后，系统按用途进入对应处理链路。
- 被忽略文件不进入后续解析任务，但保留导入日志。

### 5.3 单文件分流规则

系统按 `file_purpose` 分流：

- actual_bid：进入 Document 解析、Bid Outline 抽取和 Candidate Knowledge 生成任务。
- template_file：进入 Template / Template Chapter / Template Material 解析任务，可同时生成 Candidate Knowledge。
- qualification：进入 Manual Asset 候选流程。
- ppt_material、cover_guide、writing_guide：进入 Template Material 或素材元数据管理流程。
- wiki_source：进入 Wiki 候选流程。
- other：保留为附件或忽略，不自动进入知识生产。

### 5.4 去重与重导入

去重规则：

- 优先使用 `file_hash` 判断重复文件。
- 若无法计算 hash，则使用 `file_name + file_size` 作为辅助判断。
- 重复文件默认不重复解析，但允许用户选择“作为新版本导入”。

重导入规则：

- 未处理或处理失败的 File Import 可重新处理。
- 已生成但未发布的 Candidate Knowledge 可删除或标记 rejected。
- 已发布 KU、Wiki、Template、Manual Asset 不允许物理删除，只能走废弃流程。

## 6. 核心对象模型

### 6.1 Document

Document 沿用 V2.0 定义，V3.0 新增来源字段。

新增字段：

```json
{
  "import_id": "",
  "source_type": "actual_bid | template_doc | qualification_doc | ppt_material | wiki_source | other",
  "source_usage": "knowledge_extract | template_parse | asset_manage | reference_only",
  "product_category_ids": [],
  "bid_project_name": "",
  "bid_customer_name": ""
}
```

要求：

- 实际标书导入时 `source_type = actual_bid`。
- 模板文档导入时 `source_type = template_doc`。
- 资质材料导入时优先进入 Manual Asset 流程。
- Document 解析后可同时产生 Document Tree、Bid Outline 和 Candidate Knowledge。

### 6.2 Document Tree Node

Document Tree Node 沿用 V2.0，新增与目录沉淀相关字段：

```json
{
  "chapter_taxonomy_id": "",
  "product_category_ids": [],
  "is_outline_node": true,
  "candidate_template_chapter_id": "",
  "candidate_pattern_id": ""
}
```

要求：

- 实际标书中的章节节点可被映射到 Chapter Taxonomy。
- 高价值章节节点可生成 Candidate Knowledge。
- 目录节点可生成 Bid Outline Node。

### 6.3 Knowledge Unit

KU 沿用 V2.0，新增产品分类和来源批次字段。

新增字段：

```json
{
  "product_category_ids": [],
  "chapter_taxonomy_id": "",
  "import_id": "",
  "candidate_id": "",
  "template_library_id": "",
  "bid_outline_id": ""
}
```

要求：

- 从实际标书提取的 KU 必须标记来源项目和来源目录。
- 从模板库提取的 KU 必须标记模板库来源。
- 人工确认时必须确认 product_category 和 chapter_taxonomy。

### 6.4 Template Library

Template Library 表示一组相关标书模板集合。

字段模型：

```json
{
  "template_library_id": "",
  "kb_id": "",
  "library_name": "",
  "library_type": "technical | commercial | qualification | product_specific | custom",
  "source_import_id": "",
  "product_category_ids": [],
  "owner": "",
  "source_author": "",
  "source_updated_time": "",
  "status": "draft | reviewing | published | deprecated | archived",
  "version": "1.0",
  "created_time": "",
  "updated_time": ""
}
```

要求：

- 一个模板库可包含多个 Template。
- 模板库不从文件夹自动生成，可由用户在单文件导入时选择或手工创建。
- 模板库发布后才参与模板推荐。
- 模板库保留关联导入文件、作者、更新时间和版本信息。

### 6.5 Template

Template 是可实例化的标书结构模板。

字段模型：

```json
{
  "template_id": "",
  "template_library_id": "",
  "kb_id": "",
  "template_name": "",
  "template_type": "technical_bid | commercial_bid | qualification | chapter_set | custom",
  "product_category_ids": [],
  "applicable_project_types": [],
  "applicable_customer_types": [],
  "status": "draft | reviewing | published | deprecated | archived",
  "version": "1.0"
}
```

要求：

- Template 不等同于单个 docx 文件，而是可实例化结构。
- Template 下包含多个 Template Chapter。
- Template 可由模板库解析生成，也可由 Bid Outline 转换生成。

### 6.6 Template Chapter

Template Chapter 表示模板中的章节节点。

字段模型：

```json
{
  "template_chapter_id": "",
  "template_id": "",
  "parent_id": "",
  "title": "",
  "level": 1,
  "sort_order": 0,
  "chapter_taxonomy_id": "",
  "product_category_ids": [],
  "expected_knowledge_types": [],
  "bound_wiki_ids": [],
  "bound_ku_ids": [],
  "bound_material_ids": [],
  "variable_ids": [],
  "rule_ids": [],
  "required": true,
  "status": "draft | published | deprecated"
}
```

规则类型：

- required：必选章节。
- optional：可选章节。
- conditional：条件章节。
- mutex：互斥章节。
- product_match：指定产品分类下启用。
- asset_required：需要特定资质或素材。

### 6.7 Template Variable

Template Variable 表示模块组织建议或高级模板实例中需要替换的变量。

字段模型：

```json
{
  "variable_id": "",
  "template_id": "",
  "template_chapter_id": "",
  "variable_key": "project_name",
  "display_name": "项目名称",
  "value_type": "string | number | date | enum | text",
  "required": true,
  "default_value": "",
  "options": [],
  "description": ""
}
```

要求：

- V3.0-MVP 只支持简单占位符替换，例如 `{{project_name}}`。
- 不支持复杂表达式和脚本计算。
- 变量替换结果必须记录在 Template Instance 和 Generation Snapshot 中。

### 6.8 Template Rule

Template Rule 表示模板章节的启用、互斥、必选、素材依赖等规则。

字段模型：

```json
{
  "rule_id": "",
  "template_id": "",
  "template_chapter_id": "",
  "rule_type": "required | optional | conditional | mutex | product_match | asset_required",
  "condition": {
    "field": "product_category",
    "operator": "in | equals | contains | exists",

  },
  "action": "enable | disable | warn | require_asset",
  "message": "",
  "status": "active | inactive"
}
```

要求：

- V3.0-MVP 仅要求支持 required、optional、product_match。

### 6.9 Template Material

字段模型：

### 6.12 Document Tree 与 Bid Outline 关系

Document Tree 和 Bid Outline 是双轨对象：

- Document Tree 表示文档全文结构，用于正文解析、知识提取和来源追溯。
- Bid Outline 表示可复用的标书目录结构，用于目录匹配、模板推荐和章节模式沉淀。
- 首次从实际标书导入时，Bid Outline Node 默认由 Document Tree Node 中的标题节点生成，并通过 `source_node_id` 关联。
- 用户编辑 Bid Outline 不自动修改 Document Tree；用户编辑 Document Tree 也不自动覆盖已确认 Bid Outline。
- 当文档重新解析产生新的 Document Tree 时，系统只生成 Bid Outline 差异建议，由人工选择是否同步。
- KU 来源追溯仍以 Document Tree Node 为准；目录匹配以 Bid Outline Node 为准。

## 7. 模板文件导入与解析

### 7.1 导入流程

```text
上传单个模板文件
  → 创建 File Import
  → 自动识别文件用途、产品分类和章节类型
  → 人工确认文件用途
  → 解析模板文件
  → 生成 Template / Template Chapter / Template Material / Candidate Knowledge
```

### 7.2 模板文件识别规则

系统根据文件名、文件内容和标题结构识别可能对象：

| 识别依据 | 识别结果 |
|----------|----------|
| 文件用途确认为 `template_file` | 进入模板文件解析 |
| 文件名包含产品名，如餐补、礼包、体检 | 建议 Product Category |
| 文件名或标题包含章节名，如售后服务、供应链、系统对接 | 建议 Chapter Taxonomy |
| 文档内标题结构 | 生成 Template Chapter |
| 文档内固定段落、表格、图片 | 生成 Template Material 或 Candidate Knowledge |

归类规则：

- 用户可将单个模板文件归属到一个 Template Library。
- 如果没有选择 Template Library，系统可自动创建“未归类模板”作为默认归类。
- Template Library 只是归类标签，不由文件夹自动生成。

### 7.3 文件名识别规则

系统根据文件名识别章节顺序和类型：

- 前缀数字用于章节排序，例如 `12.企业综合实力...docx`。
- 关键词用于章节类型识别，例如“售后服务方案”“资金风险管控方案”。
- 文件扩展名用于解析策略选择。

### 7.4 人工确认

模板文件解析结果必须进入人工确认界面。

确认项：

- Template Library 归类。
- 产品分类。
- Template Chapter 层级。
- 章节类型。
- 是否作为固定模板章节。
- 是否提取为 KU 或 Wiki。
- 是否忽略。

## 8. 实际标书导入与知识沉淀

### 8.1 导入流程

```text
上传实际标书
  → Document 解析
  → Document Tree 生成
  → Bid Outline 抽取
  → Candidate Knowledge 生成
  → 人工确认
  → 发布 KU / Wiki / Chapter Pattern / Template Draft
```

### 8.2 Bid Outline 抽取

系统从实际标书中抽取目录：

- 优先使用文档内置目录。
- 其次使用标题样式和编号规则。
- 支持人工修正目录。
- 支持将目录节点映射到 Chapter Taxonomy。

验收标准：

- 一份实际标书导入后，能生成可编辑 Bid Outline。
- 用户可确认目录节点的章节类型和产品分类。
- 确认后的目录可用于目录级检索和模块组织建议。

### 8.3 Candidate Knowledge 生成

系统从 Document Tree Node 生成候选知识：

- 技术方案章节生成方案类 KU 候选。
- 产品功能章节生成产品/能力说明类 KU 候选。
- 供应链章节生成能力说明或方案类 KU 候选。
- 企业实力、资质、荣誉章节生成资质或能力类 KU 候选。
- 稳定通用段落可推荐为 Wiki 候选。

### 8.4 人工确认入库

候选知识发布前必须确认：

- 入库对象类型。
- 标题和摘要。
- 产品分类。
- 章节类型。
- 知识类型。
- 推荐使用方式。
- 是否可检索。
- 来源链是否正确。

## 10. 目录级检索与模块组织建议

### 10.1 目录级检索对象

新增检索对象：

- Bid Outline。
- Bid Outline Node。
- Template Chapter。
- Chapter Pattern。

### 10.2 目录匹配流程

```text
输入外部招标约束中的目标目录 / 章节标题 / 评分点
  → 章节标题标准化
  → 匹配 Chapter Taxonomy
  → 检索历史 Bid Outline
  → 检索 Template Chapter
  → 检索 Chapter Pattern
  → 返回相似目录、候选模块和组织建议
```

### 10.3 匹配评分口径

V3.0-增强阶段采用可解释的规则评分，避免一开始依赖黑盒模型。

`match_score` 由以下因素加权得到：

| 因素 | 默认权重 | 说明 |
|------|----------|------|
| 产品分类匹配 | 30% | Product Category 完全一致或父子分类匹配 |
| 章节分类覆盖 | 30% | 目标目录节点映射到 Chapter Taxonomy 后，被模板章节覆盖的比例 |
| 标题语义相似 | 20% | 章节标题与 Template Chapter / Bid Outline Node 的语义相似度 |
| 层级顺序相似 | 10% | 目录层级和相对顺序相似度 |
| 可用知识覆盖 | 10% | 每个目标章节是否有可用 KU/Wiki/素材 |

`coverage_rate` 计算规则：

```text
coverage_rate = 已匹配目标章节数 / 目标章节总数
```

缺失章节判断：

- 目标目录缺少该产品分类下高频 Chapter Pattern，且该 Pattern 频次超过阈值。
- 默认阈值：同一产品分类下出现频次大于等于 3 次，或出现比例大于等于 30%。

### 10.4 模块组织建议

输入：

- 产品分类。
- 项目类型。
- 客户类型。
- 外部招标文件结构要求。
- 评分点。
- 废标项。
- 目标章节要求。

输出：

- 推荐 Template Chapter 或历史章节模块。
- 推荐 KU/Wiki/素材。
- 推荐组织顺序和组合方式。
- 与招标文件要求的覆盖度。
- 缺失章节提示。
- 可用 KU/Wiki 数量。
- 风险提示。

原则：

- 招标文件中的结构要求、评分点和废标项优先于模板库建议。
- 模板库只提供历史写法和小模块组织参考。
- 若模板章节与招标要求冲突，必须提示风险，不得自动采用模板章节。

### 10.5 章节缺失诊断

系统根据 Chapter Pattern 判断目标目录可能缺失的常见章节。

示例：

- 餐补类标书常见“供应链介绍”“覆盖地域及商户数量”“员工端使用流程”“管理后台功能介绍”。
- 如果目标目录缺少这些章节，系统可提示“建议补充”。

输出：

- 与评分点的对应关系。
- 与废标项的风险提示。

- 项目名称。

- 服务区域。

要求：

- 变量替换必须记录在生成快照中。

### 11.4 条件章节

章节可按条件启用：

- 产品分类匹配。
- 客户类型匹配。
- 招标要求包含关键词。
- 必需资质存在。
- 用户手工选择。

### 11.5 生成辅助输入升级

V3.0 章节草稿生成输入不再只有知识包，也不以模板为主输入，而是包括：

- Tender Requirement Context。
- Module Assembly Suggestion。

- Product Category。
- Bid Outline Node。
- KU/Wiki Knowledge Pack。

优先级：

```text
废标项 > 评分点 > 标书结构要求 > 用户人工选择 > 知识包 > 模板库模块建议
```

输出：

- 使用的模板库参考模块。
- 使用的变量。
- 冲突提示。
- 缺失素材提示。
- Generation Snapshot。

## 13. 检索与推荐

| 对象 | 检索类型 | 用途 |
|------|----------|------|
| KU | 语义 + 关键词 + 元数据 | 内容知识检索 |
| Wiki | 语义 + 关键词 + 元数据 | 标准内容检索 |
| Template | 元数据 + 结构匹配 | 模板推荐 |
| Template Chapter | 语义 + 章节匹配 | 章节模板推荐 |
| Bid Outline | 结构匹配 | 历史目录匹配 |

### 13.3 检索请求参数模型

平台需要支持面向不同检索请求的参数化检索。检索请求不应只有 query 文本，还应包含业务上下文、召回配置、排序配置和返回配置。

请求参数建议：

```json
{
  "query": "",
  "intent": "knowledge_lookup | material_recommend | module_suggestion | trace_lookup",
  "product_category_ids": [],
  "chapter_taxonomy_ids": [],
  "knowledge_types": [],
  "file_purposes": [],
  "tender_requirement_context": {
    "outline_title": "",
    "score_points": [],
    "rejection_clauses": []
  },
  "retrieval_options": {
    "enable_bm25": true,
    "enable_vector": true,
    "enable_rerank": true,
    "top_k": 20,
    "context_expand_depth": 1
  },
  "return_options": {
    "include_trace": true,
    "include_score_detail": true,
    "include_conflict_flags": true
  }
}
```

要求：

- 不同 intent 可使用不同召回策略和排序策略。
- 检索参数必须记录到 retrieval_trace，便于后续评测和优化。
- 返回结果必须包含 score、score_detail、命中原因和来源追溯。
- 前端或调用方可调整部分检索参数，例如 top_k、过滤条件、是否启用上下文扩展。

### 13.5 检索反馈与准确率优化闭环

平台需要支持检索结果准确率不断提升。V3.0 不要求一次性实现复杂自动学习系统，但必须预留数据采集、评测和调参闭环。

反馈数据：

- 点击。
- 采纳。
- 复制。
- 加入草稿。
- 用户标记“有用/无用”。
- 用户标记“误召回”。
- 用户标记“漏召回”，并可补充期望结果。
- 人工调整产品分类、章节类型、知识类型。

优化对象：

- 查询意图识别。
- 产品分类和章节分类建议。
- BM25 字段权重。
- 向量召回模型或 embedding 策略。
- Rerank 模型和特征。
- 知识包上下文扩展深度。
- Prompt 模板。
- 返回结果排序规则。

### 13.6 检索评测集与版本化

平台应支持构建检索评测集。

评测集字段：

```json
{
  "eval_case_id": "",
  "query": "",
  "intent": "",
  "filters": {},
  "expected_object_ids": [],
  "negative_object_ids": [],
  "product_category_ids": [],
  "chapter_taxonomy_ids": [],
  "created_from": "manual | user_feedback | production_log"
}
```

评测指标：

- Recall@K。
- Precision@K。
- MRR。
- NDCG。
- 采纳率。
- 误召回率。
- 漏召回率。
- 有来源结果占比。

要求：

- 检索策略、Prompt、Embedding、Rerank 配置均需要版本号。
- 每次检索优化前后应能在同一评测集上对比指标。
- 线上反馈可转为评测用例，但必须人工确认后进入正式评测集。

## 19. 验收用例

### 19.1 导入标书模板文件

1. 用户上传一个标书模板 docx。
2. 系统创建 File Import。
3. 系统建议文件用途为 `template_file`。
4. 用户确认产品分类和章节类型。
5. 系统解析文档标题和内容，生成 Template Chapter 或 Template Material。
6. 用户确认产品分类和章节类型。
7. 用户发布 Template 或关联到 Template Library。

### 19.2 导入实际标书

1. 用户上传一份实际标书 docx。
2. 系统解析 Document Tree。
3. 系统抽取 Bid Outline。
4. 系统从章节生成 Candidate Knowledge。
5. 用户确认“产品功能介绍”章节属于某产品分类。
6. 用户将候选知识发布为 KU。

### 19.3 产品分类驱动推荐

1. 用户选择产品分类“餐补”。
2. 用户输入目标章节“供应链介绍”。
3. 系统优先推荐餐补相关 KU、Template Chapter 和 Chapter Pattern。
4. 系统过滤掉不适用于餐补的素材。

### 19.4 外部招标约束驱动的模块建议

1. 用户输入外部招标约束，包括目标目录、评分点和废标项。
2. 系统匹配历史 Bid Outline。
3. 系统推荐相关 Template Chapter、Chapter Pattern、KU/Wiki 和素材。
4. 系统显示评分点覆盖情况、缺失素材和废标风险。

### 19.5 章节草稿辅助生成

1. 用户选择一个外部招标约束中的目标章节。
2. 系统创建 Module Assembly Suggestion。
3. 系统根据评分点推荐 KU/Wiki/历史章节模块。
4. 系统根据废标项过滤或提示高风险素材。
5. 用户人工选择采用的模块和知识。
6. 用户生成章节草稿。
7. 草稿记录招标约束、引用来源、模板库参考模块和变量值。

### 19.6 检索准确率优化

1. 用户发起一次素材推荐请求，传入 query、intent、产品分类、章节类型和外部招标约束。
2. 系统记录 retrieval_trace，包括召回参数、过滤条件、排序策略和返回结果。
3. 用户标记某条结果为“无用”或“误召回”。
4. 用户补充一条期望结果，形成“漏召回”反馈。
5. 系统将反馈进入待确认队列。
6. 人工确认后，该反馈可进入 Retrieval Eval Set。
7. 后续调整检索策略时，可用同一评测集对比 Recall@K、Precision@K、MRR、NDCG 等指标。

## 20. 默认决策与待确认事项

### 20.1 V3.0 默认决策

为便于技术方案设计，V3.0 先采用以下默认决策：

1. 素材来源：V3.0-MVP 只支持手工上传单个文件；目录导入、文件夹扫描、批量导入、网盘/API 同步作为后续扩展。
2. 产品分类：平台支持预置和人工维护；从历史标书自动归纳只作为候选建议，不自动生效。
3. 模板库元数据：必须保留作者、更新时间、历史版本和关联 File Import；不保留原始文件夹路径。
4. PPT、图片、Excel：V3.0-MVP 只做元数据、附件和适用分类管理，不进入正文语义生成。
5. 目录抽取范围：V3.0-MVP 支持从历史投标文件抽取；招标文件目录、评分点、废标项由外部输入提供，知识库只消费其结构化结果。
6. 候选知识确认：V3.0-MVP 采用单人确认 + 审计日志；双人审核作为可配置扩展。
7. 模板变量：V3.0-MVP 只支持简单占位符，不支持复杂表达式。
8. 模板效果：V3.0 记录模板使用和采纳数据；中标结果归因分析不纳入 V3.0-MVP。
9. 用户角色和权限：V3.0 暂不设计权限体系，仅保留操作日志和审计字段。
10. 检索优化：V3.0 必须记录检索参数、结果反馈和评测数据，用于持续提升结果准确率。

### 20.2 仍待业务确认

1. 首批产品分类的标准名称和层级。
2. 目录相似度阈值是否需要按产品分类单独配置。
3. 模板使用效果是否需要关联真实投标项目结果。
4. 检索反馈标签是否需要与业务团队的人工质检流程打通。

| D3 | 引入 Bid Outline | 实际标书目录独立沉淀 | 目录本身是可复用结构资产 |
| D4 | 引入 Candidate Knowledge | 导入结果先进入候选区 | 防止未确认历史内容污染正式知识库 |
| D5 | 模板库定位 | 模板库是知识来源和模块组织建议来源，不是生成主约束 | 实际标书生成的主约束来自招标文件结构要求、评分点和废标项 |
| D6 | 权限后置 | V3.0 暂不设计用户角色和权限体系 | 当前阶段先聚焦知识生产、检索和优化闭环 |
| D7 | 目录级检索 | Bid Outline/Template Chapter 可检索 | 新标书编写首先需要匹配结构，再匹配内容 |
| D8 | V3.0-MVP 边界 | Phase 1-3 为 MVP，Phase 4-5 为增强和高级能力 | 防止模板推荐和生成升级挤占基础数据治理能力 |
| D9 | 简单变量优先 | MVP 只支持占位符，不支持复杂表达式 | 降低模块建议与生成辅助复杂度，先验证业务闭环 |
| D10 | Candidate 统一建模 | 候选模板章节、候选产品分类统一放入 Candidate Knowledge | 减少候选区对象数量，统一确认工作台 |
| D11 | 招标约束外置 | 招标文件结构要求、评分点、废标项不在知识库内治理 | 这些信息属于实际投标应用输入，知识库只负责消费结构化结果 |
| D12 | 检索持续优化 | 检索请求参数、反馈和评测集作为产品能力 | 知识库价值依赖结果准确率，需要支持持续重构和调优 |
